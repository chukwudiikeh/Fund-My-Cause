# Translation Contribution Guide

Fund-My-Cause uses [next-intl](https://next-intl.dev) for internationalization. Translation files live in `apps/interface/messages/`.

## Supported Languages

| Code | Language | RTL |
|------|----------|-----|
| `en` | English  | No  |
| `es` | Español  | No  |
| `fr` | Français | No  |
| `ar` | العربية  | Yes |
| `he` | עברית    | Yes |

## File Structure

```
apps/interface/
├── messages/
│   ├── en.json   ← source of truth
│   ├── es.json
│   ├── fr.json
│   ├── ar.json
│   └── he.json
└── src/i18n/
    ├── config.ts      ← locale list, RTL locales, display names
    ├── routing.ts     ← next-intl routing (localePrefix: "as-needed")
    ├── request.ts     ← server-side message loading
    └── navigation.ts  ← typed Link/useRouter/usePathname
```

## Adding a New Language

1. **Add the locale code** to `src/i18n/config.ts`:
   ```ts
   export const locales = ["en", "es", "fr", "ar", "he", "de"] as const;
   export const localeNames: Record<Locale, string> = {
     // ...existing entries...
     de: "Deutsch",
   };
   ```
   If the language is RTL, also add it to `rtlLocales`.

2. **Create the translation file** by copying `messages/en.json` to `messages/de.json` and translating every value.

3. That's it — the middleware, layout, and language selector pick up the new locale automatically.

## Translating Strings

All translation files share the same key structure as `en.json`. Keys are namespaced by page/feature:

```json
{
  "nav": { ... },
  "home": { ... },
  "campaigns": { ... },
  "bookmarks": { ... },
  "compare": { ... },
  "notFound": { ... },
  "common": { ... }
}
```

### Interpolation

Variables use `{variableName}` syntax:

```json
"raised": "{amount} XLM raised"
```

### Plurals

Use ICU message format for plurals:

```json
"deadlineAlert": "{title} ends in {days, plural, =0 {less than a day} one {# day} other {# days}}!"
```

## Using Translations in Components

**Server components** (no `"use client"`):
```tsx
import { useTranslations } from "next-intl";

export default function MyPage() {
  const t = useTranslations("home");
  return <h1>{t("heroTitle")}</h1>;
}
```

**Client components** (`"use client"`):
```tsx
"use client";
import { useTranslations } from "next-intl";

export function MyComponent() {
  const t = useTranslations("campaigns");
  return <button>{t("filterAll")}</button>;
}
```

## RTL Support

Arabic and Hebrew automatically receive `dir="rtl"` on the `<html>` element via the locale layout. The CSS in `globals.css` handles font fallbacks for RTL locales.

When writing new components, use logical CSS properties where possible (`start`/`end` instead of `left`/`right`) so they work correctly in both LTR and RTL layouts. Tailwind's `ps-*`/`pe-*` (padding-inline-start/end) utilities are preferred over `pl-*`/`pr-*`.

## Testing Translations

Run the dev server and switch languages via the globe icon in the navbar:

```bash
cd apps/interface
npm run dev
```

Navigate to `http://localhost:3000/es`, `http://localhost:3000/fr`, etc. to preview each locale. The default locale (`en`) is served at `/` without a prefix.

## Keeping Translations in Sync

When adding new UI text:

1. Add the key and English value to `messages/en.json` first.
2. Add the same key to all other locale files with a translated value (or a `[TODO]` placeholder if you don't speak the language).
3. Open a PR — the CI will catch missing keys if you add a key-completeness check.

### Checking for Missing Keys (optional script)

```bash
node -e "
const en = require('./messages/en.json');
const flat = (obj, prefix='') => Object.entries(obj).flatMap(([k,v]) =>
  typeof v === 'object' ? flat(v, prefix+k+'.') : [prefix+k]);
const enKeys = flat(en);
['es','fr','ar','he'].forEach(lang => {
  const other = require('./messages/'+lang+'.json');
  const missing = enKeys.filter(k => {
    const parts = k.split('.');
    let o = other;
    for (const p of parts) { o = o?.[p]; }
    return o === undefined;
  });
  if (missing.length) console.log(lang+':', missing);
});
"
```
